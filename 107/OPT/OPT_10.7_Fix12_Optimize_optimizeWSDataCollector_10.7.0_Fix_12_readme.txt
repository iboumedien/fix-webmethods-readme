6.0 Cautions and Warnings

1. Attempting to apply the Optimize Fix to an installation where
 MWS is configured to use the embedded Derby database will fail.
 There will be an error reported in the Software AG Update Manager
 log. For all other supported Relational Database Management Systems
 configured for MWS, the fix can be applied successfully.

 There is no workaround at this time.


 2. Software AG does not recommend uninstalling this fix because 
 Optimize or other webMethods products might not work properly.

 To be able to restore your Optimize installation, before you install
 the fix you must back up the Optimize database and your file system.

OBE-12448 (OPT_10.7_Fix12)
webMethods Optimize uses an older version of the Jettison library, which has known security vulnerabilities.
 
The issue is resolved. webMethods Optimize now uses Jettison 1.5.4.19.

WMN-8362 (OPT_10.7_Fix12)
In webMethods Monitor, several API queries consume a lot of memory.

The issue is resolved. The following services are modified so that the wm.monitor.useExactMatch parameter in monitor.cnf applies to them.

pub.monitor.process.instance:getInstanceList
pub.monitor.process.instance:getInstanceListWithFilter
pub.monitor.process.instance:getInstanceListCustomDataSet
pub.monitor.process.instance:getInstanceListCustomData
pub.monitor.process.instance:getPagedInstanceList

If you set useExactMatch to True, your search query will return only those INSTANCEID,
PARENTINSTANCEID, and CUSTOMID results which exactly match your search term. An isExactMatch input parameter is also added to these services. isExactMatch works like the useExactMatch parameter, but takes priority over it.

New parameter reference:
  isExactMatch
  
  This parameter allows you to control whether your search query will use the UPPER and LIKE
  conditions for INSTANCEID, PARENTINSTANCEID, and CUSTOMID.
  
Valid values are:
  false - Default value for isExactMatch. The search query will use the UPPER and LIKE
  conditions for INSTANCEID, PARENTINSTANCEID, and CUSTOMID.
  true - If you set isExactMatch to True, the search query will not use the UPPER and LIKE
  conditions for INSTANCEID, PARENTINSTANCEID, and CUSTOMID. The query will return only those
  results which exactly match your search term.

In webMethods Monitor, when you terminate an Oracle Database archive service and start a new service, a message for a successful operation appears, but no data is archived.

1. Install the MON_10.7_Database_Fix3 fix.

WMN-7953 (OPT_10.7_Fix7)
With webMethods Monitor, search query performance needs to be improved.

The issue is resolved. To improve search query performance on an SQL Server database configured in Integration Server, the wm.monitor.useExactMatch parameter is added to the monitor.cnf configuration file.

New parameter reference:
  useExactMatch

  This parameter allows you to control whether a search by keyword query performed from the
  Process Instance Detail page will use the UPPER and LIKE functions for instanceID and
  parentinstanceID.

Valid values are:
  false - Default value for useExactMatch. The search query will use the UPPER and LIKE
  functions for instanceID and parentinstanceID.
  true - If you set useExactMatch to True, the search query will not use the UPPER and LIKE
  functions for instanceID and parentinstanceID. The search query will return only those results
  which exactly match your search term.

OBE-11769 (OPT_10.7_Fix2)
Analytic Engine and Web Service Data Collector cannot run behind a load balancer.

The issue is resolved. You can enable Analytic Engine or Web Service Data Collector to work behind a load balancer, by doing the following:
1. Install the OPT_10.5_Fix9 fix.
2. Open the EndpointRegistry.xml file located in:
For Analytic Engine: 
<sag_install_dir>/optimize/analysis/conf/system/
For Web Service Data Collector: 
<sag_install_dir>/optimize/dataCollector/conf/system/
3. Add a new endpoint for the load balancer configuration to each 
EndpointRegistry.xml file you want to modify:
<endpoint host="HOSTNAME" name="LB Registry" path="PATH" port="PORT" protocol="PROTOCOL"/>, 
where:
<host> is the IP address or hostname of the load balancer.
For the <name> parameter, specify "LB Registry".
<path> is the routing path for the listener.
<port> is the port on which to access the load balancer.
<protocol> is the communication protocol to use with the load balancer, for example HTTP or HTTPS.

Important: You must perform the above steps after each deployment of an 
environment configuration from the user interface, 
because the EndpointRegistry.xml file is overwritten.

Optimize fills up the dead letter queue with unprocessed JMS messages.

longer than the maximum allowed length.

wrapper.java.additional.41=-Dpublish.process.meta=false

The issue is resolved.
