1.0 Fix Name

CentraSite 10.7 Fix 19



2.0 Fix ID
INM_10.7_Fix19



3.0 Product(s)/Component(s) Affected

This fix affects CentraSite 10.7.



4.0 Requirements

CentraSite Core 10.7 Fix 19 and CentraSite Server Common 10.7 Fix 19 require each
other.

For language pack requirements, see the Globalization section of the Software AG
Empower Product Support website at
https://empower.softwareag.com/Products/Globalization/default.asp



5.0 Platform Support

CentraSite 10.7 Fix 19 is supported on the same platforms and operating systems
and with the same JREs as CentraSite 10.7.



6.0 Cautions and Warnings

None.



7.0 Fix Contents

In addition to including resolutions to product defects, a fix may also  include
enhancements to existing functionality and features.  Any  of  these  items  can
result in new or changed built-in services, APIs, or  configuration  parameters.
Review the fix contents carefully before applying the fix.

This fix is cumulative, which means that it is an aggregate  of  previous  fixes
for the release.


INM-25799
Update ApacheCommons libraries to remove duplicates

The issue is resolved.


INM-25693 (Fix 16)
Update Jackson libraries in CentraSite to the newest version

The issue is resolved.


INM-25683 (Fix 15)
Vulnerable 3rd party component jQuery used - jQuery v3.3.1

CentraSite web application uses an outdated version of jQuery which can be
vulnerable.

The issue is resolved. 


INM-25671 (Fix 14)
Unable to import the exports created from CentraSite

While importing an export created from CentraSite, the import creates a
virtual service or REST service that has existing resources missing from
them. It creates new resources.

This issue is resolved. 


INM-25617 (Fix 13)
Unable to import APIs exported from CentraSite

Importing an API that is exported from CentraSite is failing because of some
missing objects.

The issue is resolved.


INM-25583 (Fix 12)
Vulnerable third party component Log4j is used

Severe vulnerabilities affecting multiple versions (2.0 to 2.17.0) of the Apache
Log4j 2 utility have been publicly disclosed. These vulnerabilities allow
unauthenticated remote code execution.

The issue is resolved by upgrading Log4j to version 2.17.1.


INM-25583 (Fix 11)
Vulnerable third party component Log4j is used

Highly severe vulnerabilities impacting multiple versions (2.0 to 2.16.0) of the
Apache Log4j 2 utility was disclosed publicly through the project's GitHub in
December 2021. This vulnerability allows for unauthenticated remote code
execution.

The issue is resolved by upgrading Log4j to version 2.17.0.


INM-25583 (Fix 10)
Vulnerable third party component Log4j is used

A highly severe vulnerability (CVE-2021-44228), impacting multiple versions
(2.0 to 2.14.1) of the Apache Log4j 2 utility was disclosed publicly through
the project's GitHub on December 9, 2021. This vulnerability allows for
unauthenticated remote code execution.

The issue is resolved.


INM-25562 (Fix 9)
When deploying from centrasite to APIGW, the rule name is lost

Context-based policy rule names are not transferred, when you deploy a
virtual service from CentraSite to API Gateway.

The issue is resolved.


INM-25406 (Fix 7)
Unable to configure SSL when using an encrypted private key

Using an encrypted key file within an SSL configuration for the CentaSite
registry/repository the server start up results in a hang up.
The last message in the server job log is:
INOXHF8241: XML thread <n> has exited due to communication error 50

This issue is resolved.


INM-25516 (Fix 7)
Centrasite Deploy to APIG

A throttling policy with the setting "Any Consumers"
throws an error message at deployment.

The issue is resolved.


INM-25506 (Fix 7)
Unable to edit service asset in Centrasite

If an object has got a dangling reference, then it could not be saved due to
an internal error message. Now the save request should update the object.

The issue is resolved.


INM-25438 (Fix 6)
Vulnerable library replaced.


INM-25443 (Fix 6)
Incorrect HTTP Connection Timeout value appears while publishing virtual service
to API Gateway

When you create a routing policy, without explicitly setting the HTTP
Connection Timeout (seconds) value and publish it to API Gateway,
the HTTP Connection Timeout value is set to 30 seconds automatically.

The issue is resolved.


INM-25348 (Fix 5)
Performance improvements for add, delete, virtualize, and deploy large OData
services

The performance for handling large OData objects is improved.
However, it takes time to save, delete or virtualize services, because of the
huge amount of objects involved.

It is recommended to increase the Java Heap of CAST when handling large OData
services.
You can monitor the progress of CRUD action by configuring log4j and set the
package com.softwareag.centrasite.api.csom to info.

The problem of aborting internal transactions is resolved.


INM-25280 (Fix 4)
Unable to modify or delete a design time policy

When a design time policy applicable to a policy itself is activated, it cannot
be deactivated. Consider a design time policy with the scope of Policy type and
Pre-Update event type and has a policy action that can fail the update of the
policy itself when executed. After such a policy is activated, it cannot be
deactivated since the policy itself fails the update or deactivation.

The issue is resolved.


INM-25276 (Fix 4)
Virtual Service not deploying from CentraSite

If there are delays in the publishing or unpublishing virtual services, then
unexpected exceptions occur and the object status is unclear.

The issue is resolved.


INM-25248 (Fix 3)
Publishing migrated virtual service from CentraSite 10.5 to API Gateway 10.5
fails

When a throttling policy enforced on a virtual service that has an application
associated with it, which is missing, publishing the virtual service fails with
a random error message.

The issue is resolved.


INM-25154 (Fix 2)
Vulnerable 3rd Party Component xmlbeans Used

Security vulnerability exists in the third-party XmlBeans library.
Web Services Stack now uses
XmlBeans 3.1.0, where the vulnerability is fixed.

This issue is resolved.


INM-25052 (Fix 1)
Missing component links and response details

The response details and the component links of a REST Service are not visible
in the view mode.

The issue is resolved


INM-25021 (Fix 1)
text change needed while choosing from Components

Text messages are not rendered properly based on the corresponding component
window titles.

The issue is resolved.


INM-25070 (Fix 1)
Arrow Icon is display transparently

When editing a REST Service transparent arrow icon is displayed in the edit
window.

The issue is resolved.


INM-25034 (Fix 1)
Vulnerable 3rd party jackson-mapper-asl component used

Vulnerable third party library jackson-mapper-asl.jar was used in SAGCruise
demo application.

This issue is resolved.



8.0 Installation

Software AG strongly recommends you to take a backup of the internal database in
which CentraSite's registry and repository data is stored before installing this
fix.

Install the fix using  Software AG Update Manager.  For  instructions,  see  the
documentation at http://documentation.softwareag.com.


8.1 Post-Installation Steps

It may be required to  start  Software AG  Runtime  and/or  CentraSite  Registry
Repository after fix installation.

Clear browser cache before starting CentraSite Control.



9.0 Uninstallation

You cannot uninstall the fixes applied on CentraSite 10.7.



10.0 Globalization

This  fix  conforms  to the  internationalization  standards  of the  webMethods
product suite  and includes  support for  operation in  any  country, locale, or
language as specified  in the Installing  webMethods  Products guide. It was not
tested with non-English configurations and non-ASCII data. However, this fix has
no globalization impact and can be applied to systems running  in  any supported
locale or configuration.


10.1 Localization

This  fix  does not  require  an updated  language  pack. It  might  contain new
messages and these messages will appear in English.
